# 鉴权审计主报告模板

只在生成 `{project_name}_auth_audit_{YYYYMMDD_HHMMSS}.md` 时读取本文件。主报告只承载 `确认漏洞` 和 `条件成立` 的认证/授权风险详情；候选、不可确认、非漏洞、加固建议放到映射表或 README。

## 硬边界

- 固定 5 个二级章节，章节名和顺序不得改变。
- 不输出 CVE、CVSS、修复版本、输出自检、模型验收、Claude 运行状态或测试提示词。
- 数量、文件数、类数、端点数、操作数和时间必须是精确可复核值；不得写 `约`、`~`、`大约`、`60+`、`17+`、尾随加号或范围数量。无法精确统计时写“未精确统计”或“不可确认”并说明缺失证据。
- 不输出完整路由清单；完整路由放映射表。
- 不把 `待验证`、`不可确认`、`非漏洞` 写成 `### AUTH-*` 风险详情。
- 主报告中出现 Burp Suite 请求或 payload 时，该条必须是 `确认漏洞` 或 `条件成立`。
- `条件成立` 只能表示外部条件待确认；如果依赖未反编译类、父类、AOP、Interceptor、Filter、网关或运行时路由内部实现，则不得进入主报告。
- 明文密码 Cookie、浏览器“记住密码”、Cookie 属性缺失、session timeout 过长不得进入主报告；除非 Cookie 本身被服务端当作登录态并可伪造/复用。
- 每个主报告风险都必须给出可导入 Burp Repeater 的 HTTP 请求；不能写“无需 Burp 请求”。
- 主报告同一 AUTH 编号只能包含同根因、同证据等级、同修复点的入口；不能把 REST、SOAP、独立 Servlet、文件上传、长轮询等不同机制合并成一个风险。
- 指向 `*.class` 的 Servlet/Filter/Interceptor、继承父类但父类未确认的 Controller、父类/AOP 未确认入口，不得进入主报告。
- 没有达到主报告门槛时，第 4 节写“本次未形成确认漏洞或条件成立风险”，不要为了填模板制造风险。
