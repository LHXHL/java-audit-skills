# 分支条件与路径可达性

本 reference 用于记录“参数到达 sink 前需要满足哪些代码路径”。它不输出漏洞可利用性，只输出路径可达性证据。

## 1. 分支分析目标

每个命中 sink 的调用链都要回答：

- 哪些条件会让代码到达 sink。
- 哪些条件会提前退出、抛出异常或转到安全分支。
- 条件值来自用户输入、配置、数据源、环境还是服务端状态。
- 条件是否改变参数可控性。

## 2. 必查分支类型

| 类型 | 关注点 |
|------|--------|
| 空值/格式检查 | 参数为空、格式错误、类型转换失败时是否提前退出 |
| 允许/拒绝列表 | 允许值、拒绝值、过滤后的值如何影响 sink |
| 权限/状态检查 | 是否需要登录、角色、租户、业务状态；只记录条件，不自行鉴权审计 |
| 功能开关 | 配置项、系统参数或 feature flag 是否控制 sink |
| 环境分支 | 运行环境、部署路径、数据源类型差异 |
| 枚举分发 | 不同 `DISPATCH_KEY` 是否进入不同 sink |
| 异常处理 | 异常后是否继续执行、吞异常、回退到默认路径 |
| 动态调用 | 动态键如何决定目标 |

## 3. 路径状态

| 状态 | 含义 |
|------|------|
| 可达 | 已有代码证据表明满足普通请求条件即可到达 |
| 条件可达 | 需要特定参数、配置、状态、权限或环境 |
| 受限可达 | 可到达，但参数值被白名单、枚举或目录限制约束 |
| 阻断 | 分支会提前退出或拒绝，无法到达 sink |
| 未确认 | 分支依赖缺失配置、动态目标或运行时状态，静态证据不足 |

不要使用“可利用/不可利用”作为本 skill 的最终结论；那属于下游专项审计。

## 4. 分析步骤

1. 从入口开始，沿调用链记录每个条件、早退、异常处理和分发分支。
2. 对每个分支写清条件表达式和变量来源。
3. 标出 sink 在哪个分支内，或是否在所有分支后共同执行。
4. 标出提前退出点和异常路径。
5. 将路径状态反馈到参数可控性表。

## 5. 输出格式

### 5.1 分支树

```text
ENTRY_OPERATION(REQUEST_PARAM)
  ├─ GUARD_CONDITION_1 -> BLOCKED_PATH [阻断]
  ├─ GUARD_CONDITION_2 -> PARAM_REWRITE [受限]
  └─ SINK_CATEGORY [条件可达]
```

### 5.2 分支判定表

| 分支 | 条件 | 条件来源 | 结果 | 路径状态 | 对可控性的影响 |
|------|------|----------|------|----------|----------------|
| B1 | `GUARD_CONDITION` | `REQUEST_PARAM` | 提前退出 | 阻断 | 不到达 sink |
| B2 | `ALLOW_RULE` | 服务端规则 + 请求参数 | 进入 sink | 受限可达 | 仅允许值可控 |

### 5.3 路径结论

路径结论只写事实，例如：`REQUEST_PARAM` 在满足 `GUARD_CONDITION` 后到达 `SINK_CATEGORY`；否则被默认值覆盖或提前退出。调用链证据支持“受限可控”，不支持直接漏洞结论。

## 6. 常见 gotchas

- 早退后的 sink 只在通过校验时可达。
- 异常处理可能吞掉校验失败并继续使用原参数。
- 默认分支可能进入通用 sink。
- 包含、前缀、后缀判断可能是过滤，也可能是业务分发，要看分支动作。
- 角色/鉴权条件只作为路径条件记录，不得扩展成越权结论。
- 动态调用需要同时追键来源和目标映射表。
- 配置开关缺失时不要猜默认值；写“未确认”并列出需确认配置。
