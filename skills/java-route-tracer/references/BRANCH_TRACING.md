# 分支条件与路径可达性

本 reference 用于记录“参数到达 sink 前需要满足哪些代码路径”。它不输出漏洞可利用性，只输出路径可达性证据。

## 1. 分支分析目标

每个命中 sink 的调用链都要回答：

- 哪些条件会让代码到达 sink。
- 哪些条件会提前 return、throw 或转到安全分支。
- 条件值来自用户、配置、数据库、环境还是服务端状态。
- 条件是否改变参数可控性。

## 2. 必查分支类型

| 类型 | 关注点 |
|------|--------|
| 空值/格式检查 | 参数为空、格式错误、类型转换失败时是否提前退出 |
| 白名单/黑名单 | 允许值、拒绝值、过滤后的值如何影响 sink |
| 权限/状态检查 | 是否需要登录、角色、租户、业务状态；只记录条件，不自行鉴权审计 |
| 功能开关 | 配置项、系统参数、profile、feature flag 是否控制 sink |
| 环境分支 | OS、容器、JDK、部署路径、数据库类型差异 |
| switch/枚举分发 | 不同 action/type/code 是否进入不同 sink |
| 异常处理 | catch 后是否继续执行、吞异常、回退到默认路径 |
| 反射/动态调用 | methodName、className、beanName、mapper id 如何决定目标 |

## 3. 路径状态

统一使用以下状态：

| 状态 | 含义 |
|------|------|
| 可达 | 已有代码证据表明满足普通请求条件即可到达 |
| 条件可达 | 需要特定参数、配置、状态、权限或环境 |
| 受限可达 | 可到达，但参数值被白名单、枚举或目录限制约束 |
| 阻断 | 分支会 return/throw/拒绝，无法到达 sink |
| 未确认 | 分支依赖缺失配置、反射目标或运行时状态，静态证据不足 |

不要使用“可利用/不可利用”作为本 skill 的最终结论；那属于下游专项审计。

## 4. 分析步骤

1. 从入口方法开始，沿调用链记录每个 `if/else`、`switch`、三元表达式、早退、异常处理。
2. 对每个分支写清条件表达式和变量来源。
3. 标出 sink 在哪个分支内，或是否在所有分支后共同执行。
4. 标出提前退出点：`return`、`throw`、`sendError`、`redirect`、`break`、`continue`。
5. 将路径状态反馈到参数可控性表。

## 5. 输出格式

### 5.1 分支树

```markdown
method(param)
  ├─ if (param == null) -> return "error" [阻断]
  ├─ if (!allowList.contains(param)) -> param = "id" [受限]
  └─ dao.query(param) [SQL sink, 条件可达]
```

### 5.2 分支判定表

```markdown
| 分支 | 条件 | 条件来源 | 结果 | 路径状态 | 对可控性的影响 |
|------|------|----------|------|----------|----------------|
| B1 | `param == null` | HTTP 参数 | 提前返回 | 阻断 | 空值不可达 |
| B2 | `allowList.contains(param)` | 服务端白名单 + HTTP 参数 | 进入查询 | 受限可达 | 仅白名单字段可控 |
```

### 5.3 路径结论

```markdown
路径结论: `pageJson.orderBy` 只有在非空且属于 `allowedColumns` 时到达 SQL sink；否则被默认值覆盖或提前返回。调用链证据支持“受限可控”，不支持直接漏洞结论。
```

## 6. 常见 gotchas

- `if (invalid) return` 后面的 sink 只在 valid 时可达；不要忽略早退。
- `try/catch` 中 catch 可能吞掉校验失败并继续使用原参数。
- `switch(type)` 的 default 分支可能进入通用危险方法。
- `contains`/`startsWith` 可能是黑名单也可能是业务分发，要看分支动作。
- 角色/鉴权条件只作为路径条件记录，不得扩展成越权结论。
- 反射调用需要同时追字符串来源和目标映射表。
- 配置开关缺失时不要猜默认值；写“未确认”并列出需确认配置。
