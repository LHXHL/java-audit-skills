# 文件读取审计报告模板

生成 `java-file-read-audit` 报告前必须读取本文件。模板用于真实代码审计输出，不得诱导编造漏洞、PoC、CVSS、CVE、修复版本或验证成功结果。

## 硬约束

1. 报告必须且只能包含 6 个编号章节。
2. 编号章节标题必须与模板完全一致：`## 1. 审计概述`、`## 2. 结论统计`、`## 3. 文件读取操作映射`、`## 4. 候选风险与非漏洞依据`、`## 5. 风险详情`、`## 6. 审计结论`。
3. 不得新增 `## 0`、`## 7`、`## 输出自检`、`## Payload 字典` 等额外章节；额外信息必须放入模板已有章节。
4. 不得输出技能源校验、测试提示词、模型自检、Claude 运行状态或验收清单。
5. 结论状态必须使用中文枚举：确认漏洞、条件成立、待验证、不可确认、非漏洞。
6. 不得输出 CVE、CVSS、修复版本、真实敏感文件内容、漏洞利用成功或未验证系统类型。
7. 确认漏洞和条件成立项必须包含 `#### Burp Suite 请求` 与 `#### Payload`；待验证、不可确认和非漏洞项不得输出可复制请求或 payload。
8. Burp Suite 请求必须是合法原始 HTTP 请求，首行只能写路径和查询串，例如 `GET /download?file={{payload}} HTTP/1.1`。
9. 没有确认真实入口、HTTP 方法、参数名或 Content-Type 时，不得编造 Burp 请求；相关项只能留在第 4 节并写补证路径。
10. `结论统计` 数量必须与 `文件读取操作映射` 中相同状态的行数一致。
11. `文件读取操作映射` 只记录真实文件读取 sink 或候选 FILE sink；静态资源目录、框架版本、配置观察等不要作为映射行。
12. 正式报告不得输出工具权限、网络限制、命令失败、模型规则编号、测试过程、具体工具不可用信息或“根据 hard rule”等内部信息；反编译不足只写“本轮未取得关键类可读实现/方法体”。
13. 文件名和生成时间必须使用真实生成时间，不得使用 `000000`、`19700101`、占位时间或固定时间。
14. 第 5 节只写 `确认漏洞` 或 `条件成立` 风险详情；`待验证`、`不可确认`、`非漏洞` 不得在第 5 节展开详情，只能放入第 4 节表格和第 6 节补证说明。
15. 只有方法名、字段名、常量池字符串、JSP 拼接、配置映射或“未发现某校验字符串”时，不能写 `条件成立`；必须标为 `待验证` 或 `不可确认`。
16. Burp Suite 请求和 Payload 必须使用 `{{...}}` 占位符；不得写真实系统敏感路径、真实生产路径、真实敏感文件内容或验证成功结论。
17. 写完报告后的对话回复只需给出报告路径和一句简短结论；不要输出报告质量自检、审批/权限信息、内部校验脚本、额外漏洞摘要或后续工具建议。

## 文件命名

```text
{project_name}_file_read_audit_{YYYYMMDD_HHMMSS}.md
```
