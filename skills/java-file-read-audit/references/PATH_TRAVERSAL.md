# 路径遍历与防护判断

只在需要判断路径可控性、路径遍历 payload 或防护充分性时读取本文件。

## 危险模式

| 模式 | 风险判断 |
|------|----------|
| 用户输入直接作为路径 | 若入口可达且无白名单/规范化，通常可升格 |
| 固定 base + 用户 fileName | 需要 canonical/normalize 后确认未逃出 base |
| 只校验扩展名 | 只能限制文件类型，不限制目录；可能条件成立 |
| 黑名单删除 `../` | 常可被编码、双写、反斜杠绕过 |
| 只做 `startsWith(base)` | base 未规范化或缺路径边界时可能绕过 |
| 支持绝对路径 | 可直接尝试应用内测试文件或系统公共文件 |

## 充分防护

以下组合通常可判非漏洞：

- 文件 ID 映射到服务端闭合集合路径，用户不能提交路径字符串。
- `basePath` 固定且不可控。
- 对目标路径执行 `getCanonicalPath()`、`toRealPath()` 或等价规范化。
- 规范化后的路径必须位于规范化后的 base 内，并考虑路径边界。
- 文件名来自闭合白名单或严格映射。
- 入口鉴权和对象归属由 `java-auth-audit` 或上游证据确认。

## 不充分防护

- 仅 `contains("..")`。
- 仅 `replace("../", "")`。
- 仅扩展名白名单。
- 仅长度、非空、正则不含空格。
- 只检查原始字符串，不检查解码和规范化结果。
- 只检查 `/`，未检查 `\`、编码斜杠或绝对路径。

## Payload 选择原则

- 使用占位符，不写真实生产路径或真实文件内容。
- 优先使用应用内低风险测试文件占位符，例如 `{{safe_test_file}}`、`{{known_allowed_file}}`、`{{app_static_test_file}}`、`{{baseline_file}}`。
- 系统相关路径也必须写成占位符，例如 `{{linux_safe_probe}}`、`{{windows_safe_probe}}`；不得写真实系统敏感路径。
- 不输出批量字典、自动化枚举脚本或敏感文件内容。
- 预期观察只能写响应差异、拒绝访问、受控测试文件内容差异或服务端日志差异；不得写“返回真实敏感文件内容”。

## 常见 payload 类别

| 类别 | 示例 |
|------|------|
| 相对路径 | `../{{safe_test_file}}`、`../../{{safe_test_file}}` |
| URL 编码 | `..%2f..%2f{{safe_test_file}}` |
| 双重编码 | `..%252f..%252f{{safe_test_file}}` |
| 反斜杠 | `..\\..\\{{safe_test_file}}` |
| 绝对路径 | `{{absolute_safe_test_file}}` |

## Gotchas

- `normalize()` 不访问文件系统，符号链接可能仍需 `toRealPath()` 或 `getCanonicalPath()`。
- `startsWith` 要比较路径对象或带分隔符边界的 canonical 字符串。
- URL 解码可能发生在容器、框架或业务代码多层，payload 要和实际解码链匹配。
- Java 新版本不支持空字节路径截断，不要把 `%00` 当默认有效 payload。
