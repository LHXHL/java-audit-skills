# SQL 注入审计报告模板

生成 `java-sql-audit` 报告前必须读取本文件。模板用于真实代码审计输出，不得诱导编造漏洞、PoC、CVSS、CVE、修复版本或验证成功结果。

## 硬约束

1. 报告必须且只能包含 6 个编号章节。
2. 编号章节标题必须与模板完全一致：`## 1. 审计概述`、`## 2. 结论统计`、`## 3. SQL 操作映射`、`## 4. 候选风险与非漏洞依据`、`## 5. 风险详情`、`## 6. 审计结论`。
3. 不得新增 `## 0`、`## 7`、`## 输出自检`、`## 调用链表`、`## 限制与证据缺口` 等额外章节；额外信息必须放入模板已有章节。
4. 不得输出技能源校验、测试提示词、模型自检、Claude 运行状态或验收清单。
5. 所有 `【填写】` 占位符必须替换；无内容时写 `无`、`N/A` 或“未确认”。
6. 结论状态必须使用中文枚举：确认漏洞、条件成立、待验证、不可确认、非漏洞。
7. 不得输出 CVE、CVSS、修复版本、数据库返回包、漏洞利用成功或未验证的数据库类型。
8. 不得输出 DML/DDL、堆叠查询、命令执行、时间延迟、文件读写、数据外带、批量利用 payload。
9. 确认漏洞和条件成立项必须包含 `#### Burp Suite 请求` 与 `#### Payload`；待验证、不可确认和非漏洞项不得输出可复制请求或 payload。
10. Burp Suite 请求必须是合法原始 HTTP 请求，首行只能写路径和查询串，例如 `GET /path?keyword={{payload}} HTTP/1.1`，不能写 `GET {{host}}/path HTTP/1.1`。
11. 没有确认真实入口、HTTP 方法、参数名或 Content-Type 时，不得编造 Burp 请求；相关项只能保留在第 4 节并写补证路径。
12. 同根因多入口遵守 `../java-shared/VULNERABILITY_GROUPING.md`。
13. `结论统计` 数量必须与 `SQL 操作映射` 中相同状态的行数一致。
14. `SQL 操作映射` 只记录真实 SQL/HQL/JPQL/native SQL sink 或候选 SQL sink；过滤器启用状态、框架版本、配置观察、导入关系等非 SQL sink 信息不要作为映射行。
15. `SQL 操作映射` 的 `结论状态` 单元格只能填写一个中文枚举，不得写“非漏洞（部分）；待补证”这类混合状态。若同一 SQL 既有安全命名参数又有未确认动态片段，按动态片段状态写“待验证”或“不可确认”，并在第 4 节解释安全部分。
16. 所有表格序号必须从 1 开始连续递增，不得跳号。
17. 正式报告不得输出工具权限、网络限制、命令失败、模型规则编号、测试过程或“根据 hard rule”等内部信息；反编译不足只写“本轮未取得关键类可读实现/反编译方法体”。
18. SQL 报告不得列组件版本或建议扫描具体 CVE；如需依赖风险，仅写“建议交给组件扫描专项”。
19. 报告全文禁止出现三个连续英文句点；需要省略非关键字段时写“省略非关键字段”，或只保留已确认的最小片段。

## 文件命名

```text
{project_name}_sql_audit_{YYYYMMDD_HHMMSS}.md
```
