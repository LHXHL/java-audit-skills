# 反编译工具使用指南（共享）

所有审计 skill 共用本策略。反编译只用于补齐项目证据，不用于扩大范围或凭命名猜测结论。

## 1. 工具获取

若用户提供工具路径，直接使用。否则优先在输出目录查找已有工具；确实缺失时再按当前项目允许的方式获取。

```bash
find {output_path} -name "cfr-*.jar" -type f
java -jar {CFR_JAR} --help
```

## 2. 调用约束

- 工具输入是编译产物文件，不是源码目录。
- 递归反编译必须先由文件枚举得到目标清单。
- 输出写到当前 worker 或当前专项允许的 `decompiled/` 目录。
- 不得覆盖其他 worker、其他阶段或共享产物。

## 3. 最小化策略

优先只反编译和当前结论直接相关的类：

- `ENTRY_ROUTE` 或 `ENTRY_OPERATION` 对应的 handler。
- `DISPATCH_RULE`、`AUTH_GATE`、`TRACE_HELPER` 指向的关键实现。
- `SINK_CATEGORY` 对应的真实执行点。
- 参数对象、父类、接口实现、配置映射和工具封装。

不要一上来反编译整个项目，除非输入完全是编译产物且无法建立候选边界。

## 4. 批量反编译

批量任务应先由项目专用脚本生成候选文件清单，再反编译清单中的目标。

```bash
find {classes_path} -name "*.class" | xargs -L 50 java -jar {CFR_JAR} --outputdir {output_path}/decompiled
```

如果候选过多，先按 route、trace task、sink category 或 worker 范围拆分。

## 5. 源码混合策略

1. 已有可读源码时优先读取源码。
2. 源码缺失时只反编译必要编译产物。
3. 已反编译结果复用，避免重复生成。
4. 报告必须标注反编译来源和可信度。

## 6. 失败处理

反编译失败时：

- 记录产物路径、错误摘要和缺失方法。
- 从接口、父类、配置、契约或调用处补证。
- 无法补证时写 blocked，不把类名或字符串线索当结论。

## 7. 验证清单

- 类路径与候选一致。
- 方法签名可读。
- 参数类型可解析。
- 关键语句、调用边或 sink 证据存在。
- 缺失行号时使用方法级证据，不捏造源码行号。

## 8. 报告标注

```markdown
位置: 【真实项目类方法或方法级证据】
来源: 【反编译产物路径】
限制: 【行号缺失/变量名缺失/实现缺失/无】
```
