# 内部证据矩阵

本文件定义漏洞审计过程中的内部证据记录。`vulnerability-type-screening.md` 是深度审计前置的漏洞族初筛表；`VULN-CAND-xxx-evidence-matrix.md` 是具体候选进入深度审计后的证据矩阵。二者都只保留在 `workspace/evidence/`，不得复制到最终报告。

## 文件位置

漏洞审计先创建漏洞族初筛表，再为具体漏洞候选创建证据文件，均放在审计工作目录的 `evidence/` 下：

```text
workspace/evidence/vulnerability-type-screening.md
workspace/evidence/VULN-CAND-001-evidence-matrix.md
workspace/evidence/VULN-CAND-001-related-routes.md
```

## 状态标记

| 标记 | 含义 |
|---|---|
| `[x]` | 初筛表：存在明确候选，必须生成候选并深审；证据矩阵：已举证 |
| `[?]` | 初筛表：存在可疑面或证据不完整，必须生成候选并深审；证据矩阵：需要人工验证 |
| `[ ]` | 未检查，仅允许作为过程态；最终报告前不得保留 |
| `[-]` | 已检查，无候选、缺证据或不适用；必须写明依据和处理说明 |
| `[!]` | 已检查，被明确防护阻断；必须写明阻断依据和处理说明 |

## 漏洞族初筛表

初筛表必须列出 `vulnerability-hypotheses.md` 中的常见 Java 漏洞族，并允许在末尾补充项目特有业务漏洞类型。必须逐项检查并完成状态标记后，再进入深度审计。

`[x]` 只表示该漏洞族存在候选面或相关证据，需要进入深度审计；不得把它解释为确认漏洞。不要使用“注入类”“文件类”“鉴权类”等过粗合并行替代逐项检查。

初筛完成后，为每个 `[x]` 或 `[?]` 漏洞族下的具体候选创建 `VULN-CAND-xxx-evidence-matrix.md`。`[x]` 和 `[?]` 行必须填写“发现的具体候选”和“候选 ID”；不得只标记而不生成候选。

## 多候选拆分规则

- 每个 `[x]` 和 `[?]` 漏洞族必须至少生成一个候选并进入深度审计。
- 同一漏洞族下发现多个独立入口、根因或传播链时，必须拆成多个候选。
- 候选 ID 按发现顺序递增，例如 `VULN-CAND-001`、`VULN-CAND-002`。
- 初筛表的“发现的具体候选”和“候选 ID”可以列多个值；每个候选都必须有独立证据矩阵。
- 如果多个候选最终证明是同一根因，最终报告按根因归并；如果根因不同，则分别输出确认漏洞。
- `[?]` 与 `[x]` 的差别只在初筛置信度，不在是否深审；`[?]` 深审后也必须闭环为 `确认`、`降级` 或 `放弃`。
- `[-]` 和 `[!]` 不强制生成候选，但必须填写初筛依据和下一步/处理说明。
- 任一 `[x]`/`[?]` 漏洞族缺少候选 ID、缺少对应证据矩阵，或候选矩阵仍停留在 `候选` 状态时，不得生成最终报告。
- 最终报告前初筛表不得保留 `[ ]`。

## 闭环校验

生成最终报告前必须运行：

```bash
python3 skills/java-audit/scripts/validate_evidence_closure.py <workspace>
```

校验失败时，必须回到深度审计补齐候选、证据矩阵或最终状态。该脚本只校验 `[x]`/`[?]` 是否完成流程闭环，不判断漏洞是否真实成立。

## 归并规则

- 只有主入口满足确认漏洞门槛后，才进入同源路由扩展。
- 同源路由必须逐条确认；不能因为同类型、同 sink、同方法名或同 controller 直接写成确认漏洞。
- 同源路由只有同样满足入口、参数来源、鉴权方式、传播链、触发条件、payload/Burp 请求、影响成立，才允许写入确认漏洞的“其他确认受影响入口”。
- 证据不足但有人工验证价值的同源路由，写入高风险线索。
- `[!]` 被防护阻断、不可达或已放弃的同源路由只保留在 `evidence/` 内部文件，不进入最终报告。
- 主入口必须提供完整 BurpSuite 原始 HTTP 请求包；其他确认路由可以提供最小差异请求包或请求摘要，并说明如何替换路径、参数、鉴权信息和 payload。
