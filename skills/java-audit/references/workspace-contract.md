# 审计工作目录规范

## 默认目录

默认先创建 `java-audit-workspace/`。如果目标目录已存在，脚本不得默认复用，也不要追加序号；应在原目录名前增加随机前缀，例如 `a3f9c2-java-audit-workspace/`。

这个规则用于多 Agent 同时审计同一项目时隔离证据、日志、临时脚本和报告，避免多个 Agent 写入同一个目录。只有用户明确要求复用已有目录或脚本显式传入 `--reuse` 时，才允许使用原目录。

## 目录职责

| 目录 | 内容 |
|---|---|
| `tools/` | CFR 等审计工具，只放本次审计需要的工具 |
| `decompiled/` | CFR 反编译输出，按目标文件名分子目录 |
| `reports/` | 最终 Markdown 报告和报告校验结果 |
| `tmp/` | 临时清单、候选列表、中间结果 |
| `logs/` | 工具下载、反编译、校验脚本运行日志 |
| `evidence/` | 组件表、组件漏洞命中、Query Pack 命中、漏洞族初筛表、证据矩阵、关键调用链片段、HTTP 样本 |

## 硬约束

- 不要把反编译结果写回源码目录。
- 不要把下载的 CFR 放到全局路径或项目根目录。
- 不要把临时脚本、日志或草稿报告混在目标项目业务代码中。
- 不要在目录名冲突时复用已有审计工作目录；读取初始化脚本输出的 `workspace` 字段作为后续命令输入。
- 工作目录中的中间文件不能自动等同于漏洞证据；报告仍需写明文件、方法、行号或方法级证据。

## 建议命令

```bash
python3 skills/java-audit/scripts/init_audit_workspace.py --base <目标项目目录>
```

输出 JSON 中的 `workspace` 字段是后续脚本的输入。
