# 漏洞验收标准

## 确认漏洞六门槛

一个漏洞只有同时满足以下六项，才允许写入“确认漏洞”：

| 门槛 | 必须证明 |
|---|---|
| 可达 | 有明确外部入口，例如路由、Servlet、Filter、RPC/WebService 方法或可调用 Web 入口 |
| 可控 | 用户能控制关键参数，并能说明参数来源、名称、绑定方式和进入代码的位置 |
| 可传播 | 从 source 到 sink 有清晰文件/方法级链路，中间没有有效防护阻断 |
| 可利用 | sink 语义能造成真实安全影响，且项目当前防护不足 |
| 可复现 | 有 payload 和 BurpSuite Repeater 可用的原始 HTTP 请求包 |
| 影响成立 | 能说明触发后的具体影响，例如读取文件、执行 SQL、越权访问、命令执行 |

## 降级规则

出现任一情况，必须降级到“高风险线索 / 下一步人工验证”：

- 只有危险 sink，没有证明外部入口。
- 有入口，但参数是否可控不清楚。
- 调用链依靠猜测、跨缺失代码推断或只靠方法名关联。
- 鉴权、校验、编码、白名单、类型约束可能阻断利用。
- payload 无法从代码证据构造。
- 原始 HTTP 请求包需要编造路径、参数、Header、Cookie、Content-Type 或 body。
- 反编译结果缺少关键方法、父类、接口或配置，无法补证。

## 禁止确认的材料

以下材料只能作为线索，不能单独作为确认漏洞：

- 关键词、正则、grep 命中。
- 组件版本或 CVE 命中。
- 危险 import、依赖、注解或配置项。
- 只有 sink，没有 source-to-sink 传播链。
- 只有 payload 思路，没有入口和请求包。
- 只有“可能存在”“疑似”“待验证”的表述。

## 证据写法

确认漏洞中每条证据都应尽量包含：

- 文件路径和行号；反编译无行号时写类名、方法名和反编译文件路径。
- 调用链使用 `入口 -> 中间方法 -> sink` 格式。
- 防护判断写清楚“缺失/绕过/无效”的项目内证据。
- 不确定性必须显式写在限制说明或高风险线索中，不能藏在确认漏洞里。
