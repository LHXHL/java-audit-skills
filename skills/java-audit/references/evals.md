# 触发与失败样例

## 正例：应触发

| 用户输入 | 理由 |
|---|---|
| 帮我快速审计这个 Spring Boot 项目有没有可利用漏洞，给 payload 和 Burp 请求包。 | Java 目标 + 快速漏洞审计 + 复现材料 |
| 这个 war 包帮我反编译后审计一下，只报确认漏洞。 | WAR + CFR 反编译 + 确认漏洞报告 |
| 看这个 Java diff 有没有引入安全漏洞，能复现的才报。 | Java diff + 漏洞确认 |
| 帮我梳理出当前源码的路由信息，输出报告。 | Java 目标 + 路由信息梳理报告 |
| 帮我梳理出当前源码下鉴权信息。 | Java 目标 + 鉴权信息梳理报告 |
| 确认一个文件读取漏洞后，继续看下载、预览、导出接口是否也受影响。 | Java 漏洞审计 + 同源路由扩展 + 根因归并 |
| 先把 Java 常见漏洞类型列成表逐项过一遍，有候选的再深入审计。 | Java 漏洞审计 + 内部漏洞类型初筛表 + 深度审计 |
| 按常见 Java 漏洞类型逐项过一遍，只输出真实漏洞。 | Java 漏洞审计 + 漏洞族初筛前置 + 最终只输出确认漏洞 |

## 反例：不应触发

| 用户输入 | 理由 |
|---|---|
| 帮我解释这个 Controller 的业务逻辑。 | Java 解释，不是安全审计 |
| 写一篇 SQL 注入原理培训文章。 | 安全内容创作，不针对具体 Java 目标 |
| 帮我设计一套权限系统方案。 | 架构设计，不是梳理具体源码鉴权信息 |

## 边界例

| 用户输入 | 预期 |
|---|---|
| 先列一下关键路由，再看看有没有漏洞。 | 触发，先做路由证据，再进入漏洞审计 |
| 给我写 Semgrep 规则找 Java SQL 注入。 | 不触发，除非同时要求审计具体 Java 目标 |
| 这个 JAR 里有个接口疑似任意文件读，帮我确认。 | 触发，先反编译，再按六门槛确认 |
| 按常见 Java 漏洞类型帮我过一遍，只输出真实存在的漏洞。 | 触发，漏洞类型只作为内部假设库，报告只输出确认漏洞 |

## 失败案例

| 失败表现 | 正确处理 |
|---|---|
| 只发现 `Runtime.exec` 就报告 RCE。 | 降级，除非证明入口、参数、传播链、触发条件、payload 和 HTTP 请求 |
| 未看鉴权配置，直接从 sink 命中开始报漏洞。 | 不合格，漏洞审计必须先梳理鉴权方式和路由鉴权状态 |
| 环境不支持子 Agent，却写出多个子 Agent 产物。 | 不合格，不支持时按单 agent 流程执行，不能模拟 |
| 鉴权侦查未完成就启动大量 sink 追踪。 | 不合格，漏洞审计必须先完成鉴权起手 |
| 用户要求漏洞审计时输出大量路由文件但没有漏洞结论。 | 不合格，路由只服务于漏洞证明 |
| 用户要求路由报告时顺手确认未授权漏洞。 | 不合格，路由报告不做漏洞确认 |
| 用户要求鉴权梳理时把未知鉴权状态写成未授权漏洞。 | 不合格，只能写未知或待确认 |
| 找不到确认漏洞时硬写“疑似 SQL 注入”。 | 写“未发现确认漏洞”，并列高风险线索和下一步人工验证 |
| 报告按 SQL 注入、SSRF、XXE 逐项写“不存在”。 | 不合格，漏洞假设库只供内部推导，最终报告不得输出不存在或已排除漏洞清单 |
| 跳过漏洞族初筛表，直接从 `Runtime.exec`、`${}` 或文件 API 命中写漏洞。 | 不合格，必须先完成鉴权起手和漏洞族初筛，再对 `[x]` 候选深度审计 |
| 初筛表中 SQL 注入标记 `[x]`，但没有生成 `VULN-CAND-xxx` 候选矩阵。 | 不合格，每个 `[x]` 必须生成候选并进入深度审计 |
| 初筛表中某漏洞族标记 `[?]`，但只写入高风险线索，没有生成 `VULN-CAND-xxx` 候选矩阵。 | 不合格，`[?]` 也必须进入深度审计并闭环 |
| `VULN-CAND-001` 证据矩阵仍是“候选”状态就生成最终报告。 | 不合格，最终报告前候选必须闭环为确认、降级或放弃 |
| 最终初筛表仍有 `[ ]` 未检查项就生成报告。 | 不合格，`[ ]` 只允许作为过程态，最终报告前必须闭环 |
| `[-]` 或 `[!]` 行没有初筛依据或处理说明。 | 不合格，已检查但不深审的状态也必须说明原因 |
| 一个漏洞族下发现多个不同入口或传播链，只审计第一个候选。 | 不合格，同一漏洞族可产生多个 `VULN-CAND-xxx`，必须分别补证、确认或降级 |
| 只确认一个入口后不排查同 controller/service/sink 的其他路由。 | 不合格，确认根因后必须做同源路由扩展 |
| 把证据不足、待验证或被阻断的同源路由写入确认漏洞。 | 不合格，只能进入高风险线索或内部 evidence 文件 |
| 把 `workspace/evidence/` 的漏洞类型初筛表或 evidence matrix 原样贴进最终报告。 | 不合格，内部表格不进入最终报告 |
| 漏洞类型初筛表中勾选了 SQL 注入，就直接写确认漏洞。 | 不合格，初筛勾选只代表进入深度审计，不代表漏洞确认 |
