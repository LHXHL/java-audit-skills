# 触发与失败样例

## 正例：应触发

| 用户输入 | 理由 |
|---|---|
| 帮我快速审计这个 Spring Boot 项目有没有可利用漏洞，给 payload 和 Burp 请求包。 | Java 目标 + 快速漏洞审计 + 复现材料 |
| 这个 war 包帮我反编译后审计一下，只报确认漏洞。 | WAR + CFR 反编译 + 确认漏洞报告 |
| 看这个 Java diff 有没有引入安全漏洞，能复现的才报。 | Java diff + 漏洞确认 |
| 帮我梳理出当前源码的路由信息，输出报告。 | Java 目标 + 路由信息梳理报告 |
| 帮我梳理出当前源码下鉴权信息。 | Java 目标 + 鉴权信息梳理报告 |

## 反例：不应触发

| 用户输入 | 理由 |
|---|---|
| 帮我解释这个 Controller 的业务逻辑。 | Java 解释，不是安全审计 |
| 写一篇 SQL 注入原理培训文章。 | 安全内容创作，不针对具体 Java 目标 |
| 帮我设计一套权限系统方案。 | 架构设计，不是梳理具体源码鉴权信息 |

## 边界例

| 用户输入 | 预期 |
|---|---|
| 先列一下关键路由，再看看有没有漏洞。 | 触发，先做路由证据，再进入漏洞审计 |
| 给我写 Semgrep 规则找 Java SQL 注入。 | 不触发，除非同时要求审计具体 Java 目标 |
| 这个 JAR 里有个接口疑似任意文件读，帮我确认。 | 触发，先反编译，再按六门槛确认 |

## 失败案例

| 失败表现 | 正确处理 |
|---|---|
| 只发现 `Runtime.exec` 就报告 RCE。 | 降级，除非证明入口、参数、传播链、触发条件、payload 和 HTTP 请求 |
| 用户要求漏洞审计时输出大量路由文件但没有漏洞结论。 | 不合格，路由只服务于漏洞证明 |
| 用户要求路由报告时顺手确认未授权漏洞。 | 不合格，路由报告不做漏洞确认 |
| 用户要求鉴权梳理时把未知鉴权状态写成未授权漏洞。 | 不合格，只能写未知或待确认 |
| 找不到确认漏洞时硬写“疑似 SQL 注入”。 | 写“未发现确认漏洞”，并列高风险线索和下一步人工验证 |
