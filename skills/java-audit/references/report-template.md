# Markdown 报告模板

最终报告必须使用以下章节顺序。没有确认漏洞时，第 2 节写“未发现确认漏洞”，不要删除章节。报告不得列出“已排除/不存在”的漏洞类型。

```markdown
# 【项目名称】Java 快速漏洞审计报告

生成时间: 【YYYY-MM-DD HH:MM:SS】
审计目标: 【路径或目标说明】
输入类型: 【源码/JAR/WAR/class/反编译产物/diff/混合】
审计工作目录: 【java-audit-workspace 路径】

## 1. 审计概述

| 项目 | 内容 |
|---|---|
| 审计范围 | 【范围】 |
| 主要框架/组件 | 【框架、构建工具、关键组件】 |
| 使用材料 | 【源码/反编译结果/diff/配置等】 |
| 反编译状态 | 【未使用/已完成/失败/部分完成】 |
| 鉴权起手结论 | 【主要鉴权方式、放行规则、权限边界和未知项摘要】 |
| 结论摘要 | 【确认漏洞数量与总体说明】 |

## 2. 确认漏洞

- 严重等级: 【Critical/High/Medium/Low + 理由】
- 根因: 【漏洞根因，说明为何按同一 VULN 归并】
- 受影响入口: 【主确认入口，HTTP 方法 + 路由/入口】
- 是否需要鉴权: 【需要/不需要/未知 + 判定依据】
- 鉴权方式: 【Session/JWT/Cookie/Basic/签名/无/其他 + 证据位置；未知时说明缺失材料】
- 参数来源: 【参数名、来源、绑定方式】
- Source-to-Sink 传播链: 【入口 -> 方法 -> sink】
- Sink: 【危险调用及位置】
- 触发条件: 【利用需要满足的条件】
- 防护判断: 【缺失/绕过/无效的鉴权、校验、过滤、白名单等】

- 修复建议: 【针对根因的修复建议】

## 填写规则

- 不得保留 `【...】` 占位符。
- “确认漏洞”只能放满足六门槛的漏洞。
- 确认漏洞按根因归并；不要因为多个路由受影响就拆成多个重复 VULN。
- 每个确认漏洞必须说明是否需要鉴权，以及鉴权方式或无法确认的原因。
- 主确认入口必须提供完整 BurpSuite 原始 HTTP 请求包。
- “其他确认受影响入口”只能列严格确认的同源路由；证据不足、待验证、被阻断或不可达的路由写入高风险线索或内部 evidence 文件。
- 高风险线索不得附带编造的 Burp 请求包。
- 不得把 `workspace/evidence/` 的组件暴露面表、组件漏洞命中表、Query Pack 命中表、漏洞族初筛表、`VULN-CAND-xxx` 证据矩阵或同源路由排查表复制到最终报告。
- 不得输出漏洞类型排除清单，例如“SQL 注入不存在”“SSRF 已排除”“未发现以下漏洞类型”。
- 原始 HTTP 请求必须使用 `http` fenced code block。
