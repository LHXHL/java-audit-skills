# Markdown 报告模板

最终报告必须使用以下章节顺序。没有确认漏洞时，第 2 节写“未发现确认漏洞”，不要删除章节。

```markdown
# 【项目名称】Java 快速漏洞审计报告

生成时间: 【YYYY-MM-DD HH:MM:SS】
审计目标: 【路径或目标说明】
输入类型: 【源码/JAR/WAR/class/反编译产物/diff/混合】
审计工作目录: 【java-audit-workspace 路径】

## 1. 审计概述

| 项目 | 内容 |
|---|---|
| 审计范围 | 【范围】 |
| 主要框架/组件 | 【框架、构建工具、关键组件】 |
| 使用材料 | 【源码/反编译结果/diff/配置等】 |
| 反编译状态 | 【未使用/已完成/失败/部分完成】 |
| 鉴权起手结论 | 【主要鉴权方式、放行规则、权限边界和未知项摘要】 |
| 结论摘要 | 【确认漏洞数量与总体说明】 |

## 2. 确认漏洞

<!-- 没有确认漏洞时写：未发现确认漏洞。 -->

### VULN-001 【漏洞标题】

- 严重等级: 【Critical/High/Medium/Low + 理由】
- 受影响入口: 【HTTP 方法 + 路由/入口】
- 是否需要鉴权: 【需要/不需要/未知 + 判定依据】
- 鉴权方式: 【Session/JWT/Cookie/Basic/签名/无/其他 + 证据位置；未知时说明缺失材料】
- 参数来源: 【参数名、来源、绑定方式】
- Source-to-Sink 传播链: 【入口 -> 方法 -> sink】
- Sink: 【危险调用及位置】
- 触发条件: 【利用需要满足的条件】
- 防护判断: 【缺失/绕过/无效的鉴权、校验、过滤、白名单等】
- Payload: 【可复现 payload 或 payload 形态】
- 影响: 【具体安全影响】
- 证据: 【文件:行号 或 反编译文件 + 方法级证据】
- 修复建议: 【针对根因的修复建议】

```http
POST /example HTTP/1.1
Host: example.com
Content-Type: application/x-www-form-urlencoded

param=payload
```

## 3. 高风险线索 / 下一步人工验证

| 编号 | 线索 | 当前证据 | 缺失证据 | 下一步 |
|---|---|---|---|---|
| LEAD-001 | 【线索】 | 【已有证据】 | 【缺失证据】 | 【下一步验证】 |

## 4. 覆盖范围与限制

- 已覆盖: 【文件、模块、入口或反编译范围】
- 未覆盖: 【未覆盖范围及原因】
- 工具限制: 【CFR/反编译/构建/依赖解析限制】
- 安全边界: 未对真实线上目标执行未授权请求或破坏性 payload。
```

## 填写规则

- 不得保留 `【...】` 占位符。
- “确认漏洞”只能放满足六门槛的漏洞。
- 每个确认漏洞必须说明是否需要鉴权，以及鉴权方式或无法确认的原因。
- 高风险线索不得附带编造的 Burp 请求包。
- 原始 HTTP 请求必须使用 `http` fenced code block。
