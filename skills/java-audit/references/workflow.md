# Java 快速漏洞审计工作流

## 1. 输入识别

先判断输入属于哪类材料：

- 源码目录：优先读取 `pom.xml`、`build.gradle`、`src/main`、配置文件、Controller、Filter、Interceptor、DAO、Service。
- JAR/WAR/class：先进入反编译流程，再审计反编译结果；不要只看文件名或包名下结论。
- 反编译产物：标记来源和可信度，注意变量名、行号、泛型、lambda、内部类可能不完整。
- diff：先定位新增/修改入口、危险调用、鉴权/过滤变化，再追受影响调用链。
- 混合材料：源码优先，反编译结果只补缺失类、依赖实现或部署包特有代码。

同时判断用户要的产物类型：

- 漏洞审计报告：找漏洞、确认漏洞、payload、BurpSuite 请求包、可利用链路。
- 路由信息梳理报告：梳理源码路由、接口入口、HTTP 方法、Handler、参数和证据位置。
- 鉴权信息梳理报告：梳理认证机制、授权机制、权限配置、过滤器/拦截器、路由鉴权映射。

## 2. 工程准备

审计开始前必须创建工作目录，并把所有产物放进去：

```text
java-audit-workspace/
├── tools/
├── decompiled/
├── reports/
├── tmp/
├── logs/
└── evidence/
```

不要把工具、临时脚本、反编译源码、报告草稿散落在项目根目录。如果默认目录已存在，初始化脚本会创建随机前缀的新目录；后续所有脚本都必须使用初始化输出 JSON 中的 `workspace` 字段，不要手写固定路径。

## 3. 子 Agent 启动条件

如果当前环境支持创建子 Agent，可以在工程准备完成后使用子 Agent。不要把子 Agent 写成硬依赖；环境不支持时，由当前 agent 按同一流程完成，不要伪造多 agent 产物。

启动子 Agent 前必须满足：

- 审计工作目录已创建，并已记录初始化脚本返回的实际 `workspace` 路径。
- 输入类型已识别。
- 必要反编译已完成或已记录反编译阻塞原因。
- 任务类型已确定为漏洞审计、路由梳理或鉴权梳理。
- 每个子 Agent 的输入范围、输出文件和验收标准已写清。

适合启动子 Agent 的场景：

- 多模块或多包结构，入口数量较多。
- 鉴权链复杂，存在多个过滤器、拦截器、安全配置或权限注解体系。
- 同时存在多个高价值 source/sink 候选，需要并行追踪。
- 需要独立 QA 复核确认漏洞证据链。

不适合启动子 Agent 的场景：

- 单接口、单文件、小范围 diff 或用户只问一个具体点。
- 当前环境没有真实子 Agent 能力。
- 任务只是 Java 解释、报告润色或规则编写。
- 子 Agent 只能复述主 Agent 上下文，无法产出独立证据。

漏洞审计的推荐启动顺序：

1. `auth-recon`：先梳理认证方式、鉴权位置、放行规则、权限边界和路由鉴权映射。
2. `route-surface`：在鉴权侦查后补充入口、参数、Handler 和路由证据。
3. `component-surface`：识别 Java Web 组件、配置、版本来源和实际使用点，并映射到漏洞族候选。
4. `query-pack`：基于反编译产物和源码运行固定 rg 查询集，输出并归类 search hits。
5. `sink-candidate`：基于鉴权面、组件面、Query Pack 命中和入口面提取危险调用与 source/sink 候选。
6. `trace`：只追踪高价值候选，证明 source-to-sink 传播链。
7. `qa`：独立复核确认漏洞是否满足六门槛、鉴权字段、payload 和 BurpSuite 原始请求包。

禁止把子 Agent 输出的线索直接写成确认漏洞。确认漏洞仍必须由主流程统一套用验收标准，并在报告中保留证据来源。

## 4. 任务分流与初筛证据来源

先确认当前任务是漏洞审计、路由梳理还是鉴权梳理。

如果用户明确要求路由信息梳理，路由报告就是主交付物，可以系统整理入口，但每条路由都必须有源码、配置或反编译证据。

如果用户明确要求鉴权信息梳理，鉴权报告就是主交付物，可以系统整理认证授权机制和路由鉴权映射，但不得把未知状态写成漏洞。

如果任务是漏洞审计，不要把攻击面浏览当成深度审计前的替代步骤。以下证据只用于支撑鉴权起手和漏洞族初筛：

- 对外入口：HTTP 路由、Servlet、Filter、Interceptor、WebService、RPC、消息消费者、计划任务。
- 用户输入：请求参数、JSON/XML body、文件上传、Header、Cookie、路径变量、反序列化输入。
- 危险面：数据库、文件、命令、网络请求、XML 解析、模板/表达式、对象反序列化、鉴权/授权逻辑。
- 配置：安全框架、过滤器链、拦截器、数据源、上传大小、路径映射、反序列化白名单。
- 组件面：Java Web 框架、安全框架、序列化库、ORM、RPC、模板、日志、调试端点、中间件客户端、容器组件。

只收集能服务漏洞判断的入口和证据；除非用户明确要求路由报告，否则不要输出全量路由索引作为主交付物。

## 5. 漏洞审计的鉴权起手

如果任务是漏洞审计，第一步必须先看鉴权方式，然后再决定从哪里下手。至少确认：

- 认证方式：Session、JWT、Cookie、Basic、签名、网关鉴权、Shiro、Spring Security、自定义 token 或未知。
- 鉴权位置：Filter、Interceptor、SecurityFilterChain、WebSecurityConfigurerAdapter、Shiro 配置、注解、AOP、网关配置或业务代码判断。
- 放行规则：白名单路径、静态资源、登录注册、健康检查、错误页、文档接口、条件放行。
- 路由鉴权映射：哪些入口需要登录、哪些需要角色/权限、哪些未知、哪些可能仅对象级鉴权。
- 权限边界：普通用户/管理员/租户/组织/数据归属等对象级权限判断。

从鉴权面选择审计切入点：

- 未鉴权或放行入口优先看高危 sink、敏感数据、文件/命令/SSRF/XML/反序列化等链路。
- 已鉴权入口优先看普通用户可达但影响高的功能、水平/垂直越权、对象级权限缺失、鉴权后危险操作。
- 权限未知入口先补证配置、父类、拦截器注册和反编译缺失代码；不能默认无需鉴权。
- 发现鉴权绕过线索时，先证明路由解析、过滤器顺序、匹配规则或注解覆盖，再进入漏洞确认。

不要跳过鉴权直接从 sink 命中下结论。鉴权状态会影响漏洞成立时，必须写入确认漏洞字段；无法判断且影响利用时，降级为高风险线索。

## 6. Java Web 组件暴露面识别

漏洞审计时读取 `java-web-components.md`，在漏洞族初筛前创建内部组件表，保存到 `workspace/evidence/component-surface.md`。

组件表用于识别项目实际使用到的 Java Web 框架、安全框架、序列化库、ORM、RPC、模板、日志、管理端点、中间件客户端和容器组件。识别依据优先级为依赖清单、`WEB-INF/lib`、配置文件、import、注解、反编译业务代码、shaded 业务包和实际使用点。

不要把组件当成漏洞族，也不要把组件版本、危险依赖、危险 import 或 CVE 命中直接写成确认漏洞。组件命中只用于驱动漏洞族初筛和 `VULN-CAND` 候选生成。

组件表必须完成状态闭环后再进入报告生成：

- `[x]`：存在明确组件暴露面和使用点，必须映射到漏洞族初筛并生成候选深审。
- `[?]`：组件或使用点证据不完整但存在可疑面，必须映射到漏洞族初筛并生成候选深审。
- `[-]` / `[!]`：不生成候选，但必须写清依据和处理说明。
- `[ ]`：仅允许作为过程态，最终报告前不得保留。

组件表中任一 `[x]` 或 `[?]` 行，必须填写证据位置、关联漏洞族和候选 ID；同一候选 ID 必须出现在 `vulnerability-type-screening.md` 对应漏洞族行中，并存在独立候选证据矩阵。

不反编译所有第三方依赖；只有业务代码、shaded 包、配置或实际使用点需要理解依赖实现时，才按需反编译相关类。

## 7. Query Pack 检索

漏洞审计时读取 `discovery-query-pack.md`，在组件暴露面识别后、漏洞族初筛前运行：

```bash
python3 skills/java-audit/scripts/run_discovery_queries.py --workspace <workspace> [--source <源码或反编译目录>]
```

默认检索 `<workspace>/decompiled`、`src`、`source`、`sources`；若业务代码在额外目录、业务 JAR/class 解包目录或手工反编译目录中，必须用 `--source` 显式加入。脚本使用 `rg -n -a -i`，不把 NG/语义检索作为硬流程。

Query Pack 输出到 `workspace/evidence/search-hits/`。每条命中只是候选线索，不是漏洞结论；必须完成归类和处理：

- 高价值命中生成新 `VULN-CAND-xxx`，或合并到已有候选。
- 低价值、误报、不适用或被防护阻断的命中，必须写明处理说明。
- 一个命中可映射多个漏洞族，不能因为它已被一个漏洞族处理，就忽略另一个实际相关的漏洞族。
- 同一漏洞族下多个独立入口、root cause、sink 或传播链，必须拆分为多个候选，或明确合并依据。

最终报告前 `search-hits/` 中不得存在 `未处理`、`待归类` 或空处理状态。不得把 Query Pack 原始命中表复制进最终报告。

## 8. 漏洞族初筛

漏洞审计时读取 `vulnerability-hypotheses.md` 和 `evidence-matrix.md`，先将漏洞假设库中的漏洞族列成内部漏洞族初筛表，保存到 `workspace/evidence/vulnerability-type-screening.md`。

初筛表是深度审计前置步骤。必须按 `vulnerability-hypotheses.md` 的常见 Java 漏洞族逐项检查，并吸收 `component-surface.md` 中 `[x]` / `[?]` 组件映射出的漏洞族和 `search-hits/` 中的 Query Pack 命中。不能用“注入类”“文件类”“鉴权类”等过粗分类替代，也不能只检查模型第一眼命中的 sink。

逐项检查每个漏洞族是否存在候选面或相关代码证据，例如入口、参数来源、危险 API、配置弱点、框架特征、Query Pack 命中或业务流程。`[x]` 只表示“该漏洞族存在候选面，进入深度审计”，不表示漏洞确认存在。

初筛表中未勾选、被防护阻断或与项目技术栈无关的类型只保留在内部表；不得把“已排除漏洞类型”“某漏洞不存在”“未发现以下漏洞”写进最终报告。

初筛完成后，只对 `[x]` 和 `[?]` 漏洞族下的具体候选进行深度审计。每个 `[x]` 或 `[?]` 漏洞族必须生成至少一个 `VULN-CAND-xxx` 候选，并在初筛表的“发现的具体候选”和“候选 ID”中记录；不能只标记不深审。

同一漏洞族下发现多个独立入口、根因或传播链时，必须拆成多个候选并分配 `VULN-CAND-001`、`VULN-CAND-002` 等 ID；不要因为同属一个漏洞族就只审计其中一个。

`[?]` 与 `[x]` 的差别只在初筛置信度，不在是否深审。`[?]` 深审后可以确认、降级或放弃；如果降级且有人工验证价值，再进入最终报告高风险线索。

对每个具体候选，在 `workspace/evidence/` 创建并同步维护独立候选证据矩阵。候选最终状态必须闭环为 `确认`、`降级` 或 `放弃`；状态为 `降级` 或 `放弃` 时必须写清原因。任何 `[x]`/`[?]` 漏洞族存在候选缺失、矩阵缺失或状态仍为 `候选` 时，不得生成最终报告。

`[-]` 或 `[!]` 不强制生成候选，但必须写清初筛依据和下一步/处理说明；依据应说明组件表和 Query Pack 覆盖情况，不能只写“第一眼未见”。最终报告前初筛表不得保留 `[ ]`。

生成最终报告前必须运行 `scripts/validate_evidence_closure.py <workspace>`。该脚本会同时校验组件表、Query Pack 命中、漏洞族初筛表和候选证据矩阵的流程闭环，不代表漏洞确认；通过后再进入报告生成与 `validate_report.py` 校验。

## 9. Source-to-Sink 追踪

每个候选至少追踪：

1. 外部入口和 HTTP 方法。
2. 是否需要鉴权、鉴权方式、权限要求和判定依据。
3. 用户参数如何绑定到方法参数、对象字段或上下文。
4. 参数经过的 Service、DAO、工具类、配置读取、拼接或转换。
5. 到达的真实 sink。
6. 中途存在的鉴权、校验、编码、白名单、类型转换或异常分支。

链路无法证明时，停止确认并降级为高风险线索。

追踪过程中持续更新 evidence matrix：已举证项标 `[x]`，未检查标 `[ ]`，缺证据标 `[-]`，被防护阻断标 `[!]`，需要人工验证标 `[?]`。矩阵只保留在 `workspace/evidence/`，不得复制到最终报告。

候选深审结束时必须更新矩阵的 `当前状态` 和 `决策`：确认漏洞写 `确认` 且“满足确认漏洞门槛”为“是”；证据不足但有人工验证价值写 `降级` 并说明缺失证据；被防护阻断、不可达或不成立写 `放弃` 并说明原因。

## 10. 同源路由扩展

主入口满足确认漏洞门槛后，继续排查是否存在同源路由。重点检查复用同一 controller、service、helper、mapper、sink、参数绑定模式或鉴权缺陷的入口。

每个同源路由都必须逐条套用确认漏洞门槛。只有同样证明入口、参数来源、鉴权方式、传播链、触发条件、payload/Burp 请求和影响成立，才允许并入确认漏洞的“其他确认受影响入口”。

证据不足但有验证价值的同源路由写入高风险线索；被防护阻断、不可达或已放弃的路由只记录在 `workspace/evidence/*-related-routes.md`，不进入最终报告。

## 11. 报告生成

最终报告只包含“确认漏洞”和必要的“高风险线索 / 下一步人工验证”。确认漏洞必须带 payload 和 BurpSuite 原始 HTTP 请求包；高风险线索不得伪装成确认漏洞，也不得附带编造的可复制利用材料。

确认漏洞按根因归并，而不是按路由拆散。每个 VULN 必须有主确认入口的完整 BurpSuite 原始 HTTP 请求包；其他确认受影响入口可以写最小差异请求包或请求摘要，并说明同一 payload 如何替换路径、参数、鉴权信息。

最终报告不得输出漏洞类型排除清单，也不得写“SQL 注入不存在”“SSRF 已排除”“未发现以下漏洞类型”等枚举式否定结论。找不到确认漏洞时，只写“未发现确认漏洞”；如果没有可操作线索，高风险线索章节写“无”。

路由信息梳理使用 `route-report-template.md`。鉴权信息梳理使用 `auth-report-template.md`。这两类报告不输出 payload 或 BurpSuite 请求包，除非用户随后明确切换到漏洞确认。
