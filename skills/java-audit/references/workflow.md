# Java 快速漏洞审计工作流

## 1. 输入识别

先判断输入属于哪类材料：

- 源码目录：优先读取 `pom.xml`、`build.gradle`、`src/main`、配置文件、Controller、Filter、Interceptor、DAO、Service。
- JAR/WAR/class：先进入反编译流程，再审计反编译结果；不要只看文件名或包名下结论。
- 反编译产物：标记来源和可信度，注意变量名、行号、泛型、lambda、内部类可能不完整。
- diff：先定位新增/修改入口、危险调用、鉴权/过滤变化，再追受影响调用链。
- 混合材料：源码优先，反编译结果只补缺失类、依赖实现或部署包特有代码。

同时判断用户要的产物类型：

- 漏洞审计报告：找漏洞、确认漏洞、payload、BurpSuite 请求包、可利用链路。
- 路由信息梳理报告：梳理源码路由、接口入口、HTTP 方法、Handler、参数和证据位置。
- 鉴权信息梳理报告：梳理认证机制、授权机制、权限配置、过滤器/拦截器、路由鉴权映射。

## 2. 工程准备

审计开始前必须创建工作目录，并把所有产物放进去：

```text
java-audit-workspace/
├── tools/
├── decompiled/
├── reports/
├── tmp/
├── logs/
└── evidence/
```

不要把工具、临时脚本、反编译源码、报告草稿散落在项目根目录。如果默认目录已存在，初始化脚本会创建随机前缀的新目录；后续所有脚本都必须使用初始化输出 JSON 中的 `workspace` 字段，不要手写固定路径。

## 3. 子 Agent 启动条件

如果当前环境支持创建子 Agent，可以在工程准备完成后使用子 Agent。不要把子 Agent 写成硬依赖；环境不支持时，由当前 agent 按同一流程完成，不要伪造多 agent 产物。

启动子 Agent 前必须满足：

- 审计工作目录已创建，并已记录初始化脚本返回的实际 `workspace` 路径。
- 输入类型已识别。
- 必要反编译已完成或已记录反编译阻塞原因。
- 任务类型已确定为漏洞审计、路由梳理或鉴权梳理。
- 每个子 Agent 的输入范围、输出文件和验收标准已写清。

适合启动子 Agent 的场景：

- 多模块或多包结构，入口数量较多。
- 鉴权链复杂，存在多个过滤器、拦截器、安全配置或权限注解体系。
- 同时存在多个高价值 source/sink 候选，需要并行追踪。
- 需要独立 QA 复核确认漏洞证据链。

不适合启动子 Agent 的场景：

- 单接口、单文件、小范围 diff 或用户只问一个具体点。
- 当前环境没有真实子 Agent 能力。
- 任务只是 Java 解释、报告润色或规则编写。
- 子 Agent 只能复述主 Agent 上下文，无法产出独立证据。

漏洞审计的推荐启动顺序：

1. `auth-recon`：先梳理认证方式、鉴权位置、放行规则、权限边界和路由鉴权映射。
2. `route-surface`：在鉴权侦查后补充入口、参数、Handler 和路由证据。
3. `sink-candidate`：基于鉴权面和入口面提取危险调用与 source/sink 候选。
4. `trace`：只追踪高价值候选，证明 source-to-sink 传播链。
5. `qa`：独立复核确认漏洞是否满足六门槛、鉴权字段、payload 和 BurpSuite 原始请求包。

禁止把子 Agent 输出的线索直接写成确认漏洞。确认漏洞仍必须由主流程统一套用验收标准，并在报告中保留证据来源。

## 4. 快速攻击面建立

快速浏览以下证据，决定审计方向：

- 对外入口：HTTP 路由、Servlet、Filter、Interceptor、WebService、RPC、消息消费者、计划任务。
- 用户输入：请求参数、JSON/XML body、文件上传、Header、Cookie、路径变量、反序列化输入。
- 危险面：数据库、文件、命令、网络请求、XML 解析、模板/表达式、对象反序列化、鉴权/授权逻辑。
- 配置：安全框架、过滤器链、拦截器、数据源、上传大小、路径映射、反序列化白名单。

只收集能服务漏洞判断的入口和证据；不要输出全量路由索引作为主交付物。

如果用户明确要求路由信息梳理，路由报告就是主交付物，可以系统整理入口，但每条路由都必须有源码、配置或反编译证据。

如果用户明确要求鉴权信息梳理，鉴权报告就是主交付物，可以系统整理认证授权机制和路由鉴权映射，但不得把未知状态写成漏洞。

## 5. 漏洞审计的鉴权起手

如果任务是漏洞审计，第一步必须先看鉴权方式，然后再决定从哪里下手。至少确认：

- 认证方式：Session、JWT、Cookie、Basic、签名、网关鉴权、Shiro、Spring Security、自定义 token 或未知。
- 鉴权位置：Filter、Interceptor、SecurityFilterChain、WebSecurityConfigurerAdapter、Shiro 配置、注解、AOP、网关配置或业务代码判断。
- 放行规则：白名单路径、静态资源、登录注册、健康检查、错误页、文档接口、条件放行。
- 路由鉴权映射：哪些入口需要登录、哪些需要角色/权限、哪些未知、哪些可能仅对象级鉴权。
- 权限边界：普通用户/管理员/租户/组织/数据归属等对象级权限判断。

从鉴权面选择审计切入点：

- 未鉴权或放行入口优先看高危 sink、敏感数据、文件/命令/SSRF/XML/反序列化等链路。
- 已鉴权入口优先看普通用户可达但影响高的功能、水平/垂直越权、对象级权限缺失、鉴权后危险操作。
- 权限未知入口先补证配置、父类、拦截器注册和反编译缺失代码；不能默认无需鉴权。
- 发现鉴权绕过线索时，先证明路由解析、过滤器顺序、匹配规则或注解覆盖，再进入漏洞确认。

不要跳过鉴权直接从 sink 命中下结论。鉴权状态会影响漏洞成立时，必须写入确认漏洞字段；无法判断且影响利用时，降级为高风险线索。

## 6. 候选筛选

优先选择证据强、入口明确、参数可控、sink 语义清晰的候选。不要机械按固定漏洞类型清单跑完全部项，也不要因为某类漏洞常见就忽略项目真实代码形态。

## 7. Source-to-Sink 追踪

每个候选至少追踪：

1. 外部入口和 HTTP 方法。
2. 是否需要鉴权、鉴权方式、权限要求和判定依据。
3. 用户参数如何绑定到方法参数、对象字段或上下文。
4. 参数经过的 Service、DAO、工具类、配置读取、拼接或转换。
5. 到达的真实 sink。
6. 中途存在的鉴权、校验、编码、白名单、类型转换或异常分支。

链路无法证明时，停止确认并降级为高风险线索。

## 8. 报告生成

最终报告只包含“确认漏洞”和“高风险线索 / 下一步人工验证”。确认漏洞必须带 payload 和 BurpSuite 原始 HTTP 请求包；高风险线索不得伪装成确认漏洞，也不得附带编造的可复制利用材料。

路由信息梳理使用 `route-report-template.md`。鉴权信息梳理使用 `auth-report-template.md`。这两类报告不输出 payload 或 BurpSuite 请求包，除非用户随后明确切换到漏洞确认。
