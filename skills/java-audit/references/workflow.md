# Java 快速漏洞审计工作流

## 1. 输入识别

先判断输入属于哪类材料：

- 源码目录：优先读取 `pom.xml`、`build.gradle`、`src/main`、配置文件、Controller、Filter、Interceptor、DAO、Service。
- JAR/WAR/class：先进入反编译流程，再审计反编译结果；不要只看文件名或包名下结论。
- 反编译产物：标记来源和可信度，注意变量名、行号、泛型、lambda、内部类可能不完整。
- diff：先定位新增/修改入口、危险调用、鉴权/过滤变化，再追受影响调用链。
- 混合材料：源码优先，反编译结果只补缺失类、依赖实现或部署包特有代码。

同时判断用户要的产物类型：

- 漏洞审计报告：找漏洞、确认漏洞、payload、BurpSuite 请求包、可利用链路。
- 路由信息梳理报告：梳理源码路由、接口入口、HTTP 方法、Handler、参数和证据位置。
- 鉴权信息梳理报告：梳理认证机制、授权机制、权限配置、过滤器/拦截器、路由鉴权映射。

## 2. 工程准备

审计开始前必须创建工作目录，并把所有产物放进去：

```text
java-audit-workspace/
├── tools/
├── decompiled/
├── reports/
├── tmp/
├── logs/
└── evidence/
```

不要把工具、临时脚本、反编译源码、报告草稿散落在项目根目录。

## 3. 快速攻击面建立

快速浏览以下证据，决定审计方向：

- 对外入口：HTTP 路由、Servlet、Filter、Interceptor、WebService、RPC、消息消费者、计划任务。
- 用户输入：请求参数、JSON/XML body、文件上传、Header、Cookie、路径变量、反序列化输入。
- 危险面：数据库、文件、命令、网络请求、XML 解析、模板/表达式、对象反序列化、鉴权/授权逻辑。
- 配置：安全框架、过滤器链、拦截器、数据源、上传大小、路径映射、反序列化白名单。

只收集能服务漏洞判断的入口和证据；不要输出全量路由索引作为主交付物。

如果用户明确要求路由信息梳理，路由报告就是主交付物，可以系统整理入口，但每条路由都必须有源码、配置或反编译证据。

如果用户明确要求鉴权信息梳理，鉴权报告就是主交付物，可以系统整理认证授权机制和路由鉴权映射，但不得把未知状态写成漏洞。

## 4. 候选筛选

优先选择证据强、入口明确、参数可控、sink 语义清晰的候选。不要机械按固定漏洞类型清单跑完全部项，也不要因为某类漏洞常见就忽略项目真实代码形态。

## 5. Source-to-Sink 追踪

每个候选至少追踪：

1. 外部入口和 HTTP 方法。
2. 用户参数如何绑定到方法参数、对象字段或上下文。
3. 参数经过的 Service、DAO、工具类、配置读取、拼接或转换。
4. 到达的真实 sink。
5. 中途存在的鉴权、校验、编码、白名单、类型转换或异常分支。

链路无法证明时，停止确认并降级为高风险线索。

## 6. 报告生成

最终报告只包含“确认漏洞”和“高风险线索 / 下一步人工验证”。确认漏洞必须带 payload 和 BurpSuite 原始 HTTP 请求包；高风险线索不得伪装成确认漏洞，也不得附带编造的可复制利用材料。

路由信息梳理使用 `route-report-template.md`。鉴权信息梳理使用 `auth-report-template.md`。这两类报告不输出 payload 或 BurpSuite 请求包，除非用户随后明确切换到漏洞确认。
