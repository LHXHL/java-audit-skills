# 文件上传审计报告模板

生成 `java-file-upload-audit` 报告前必须读取本文件。模板用于真实代码审计输出。对确认漏洞和条件成立风险，必须输出可提交开发单位验证的 Burp Suite 请求和 Payload；但不得诱导编造漏洞、组件漏洞编号、风险评分、修复版本、可执行后门样本、持久化内容或验证结果。

## 硬约束

1. 报告必须且只能包含 6 个编号章节；不得输出测试验收信息。
2. 编号章节标题必须完全一致：`## 1. 审计概述`、`## 2. 结论统计`、`## 3. 上传点映射`、`## 4. 候选风险与非漏洞依据`、`## 5. 风险详情`、`## 6. 审计结论`。
3. 不得新增 `## 0`、`## 7`、`## 调用链表`、`## 限制与证据缺口` 等额外编号章节。
4. 所有 `【填写】` 占位符必须替换；无内容时写 `无`、`N/A` 或“未确认”。
5. 结论状态必须使用中文枚举：确认漏洞、条件成立、待验证、不可确认、非漏洞。
6. 风险等级只使用中文：高、中、低、信息；不得输出组件漏洞编号、风险评分、修复版本或不可验证的修复版本号。
7. 上传点映射只列真实上传入口；工具类、拦截器、配置文件、静态目录和第三方库只放第 4 节依据，不计入统计。
8. `## 5. 风险详情` 只写“确认漏洞/条件成立”项；待验证、不可确认和非漏洞只放在第 3、4、6 节，不在第 5 节展开。
9. “确认漏洞/条件成立”风险详情必须包含 `#### Burp Suite 请求` 和 `#### Payload` 小节。
10. 待验证、不可确认和非漏洞项不得输出 Burp Suite 请求或 Payload。
11. Burp Suite 请求必须匹配真实入口、HTTP 方法、参数名、Content-Type、鉴权状态和必要业务字段；缺失运行值使用占位符。
12. Burp Suite 请求行必须使用 `POST /path HTTP/1.1` 格式，`Host: {{host}}` 单独作为请求头；不得把主机、协议或完整 URL 写入请求行。
13. 参数名、字段名或路径片段未确认时必须使用占位符并标注待确认，不得把推测值写成确定值。
14. Payload 必须低破坏，优先使用无害 marker 文件内容；不得输出可执行后门样本、持久化脚本、批量利用脚本、横向移动步骤，或声称上传、访问、代码执行已经达成。
15. 没有真实入口、真实 sink 或授权环境时，不得给可复制验证请求；只能写授权补证建议。
16. 不得把单纯 `maximumSize`、multipart 临时目录或容量配置加固点作为上传漏洞编号；可在第 4 节或第 6 节作为加固建议说明。
17. 不得因黑名单覆盖范围有限而列出未覆盖后缀清单；如需表达，只写“黑名单覆盖范围不足，需以白名单和内容校验为主”。
18. 报告全文禁止出现三个连续英文句点。
19. 代码中真实出现的扩展名常量可以最小引用；不得额外列举“黑名单缺少哪些后缀”、大小写绕过、双后缀绕过或编码绕过写法。
20. 文件名和生成时间必须使用真实运行时刻，不得使用 `000000`、`00:00:00`、占位时间或固定假时间。
21. 未取得反编译/反汇编结果时，报告只写“本轮未取得反编译/反汇编结果”以及因此缺少哪些代码证据；不得解释原因、环境、安装状态、权限、审批、申请、拒绝、拦截或工具许可。
22. 结论统计数量必须与“上传点映射”和“风险详情”一致；聚合时要说明哪些映射行合并到同一个风险详情。
23. 同根因多入口遵守 `../java-shared/VULNERABILITY_GROUPING.md`。

## 文件命名

```text
{project_name}_file_upload_audit_{YYYYMMDD_HHMMSS}.md
```
