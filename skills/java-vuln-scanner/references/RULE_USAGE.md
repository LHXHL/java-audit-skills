# 本地组件规则使用说明

只在解释规则数据命中结果时读取本文件。

## 规则性质

- 规则数据用于快速发现组件旧版本或已知风险区间。
- 规则命中表示 `COMPONENT_VERSION` 落入本地规则覆盖范围，不等于业务风险真实成立。
- 规则可能过期、重复、漏报或误报；正式交付要写明需复核官方公告或企业 SCA。
- 正式报告只能引用扫描脚本 JSON 中实际返回的 `RULE_ID`。

## 证据等级

| 来源 | 可信度 | 写法 |
|------|--------|------|
| 运行时依赖清单 | 较高 | 可作为运行时依赖证据 |
| 构建文件直接版本 | 较高 | 需注意 scope 和 profile |
| 父级版本管理 | 中 | 需确认子模块是否实际引入 |
| 归档文件名 | 中低 | 可能改名、重打包或合并 |
| 元数据文件 | 中低 | 可能缺失或不规范 |
| 类名/包名 | 低 | 只能说明存在代码片段，不能确定版本 |

## 去重规则

- 同一 `COMPONENT_ID + COMPONENT_VERSION + RULE_ID` 在多个模块出现时，按模块列来源，结论中聚合为一个治理项。
- 同一组件命中多个规则时，列规则命中，但不要写成多个已确认业务漏洞。
- 构建文件和运行时产物冲突时，优先标注运行时证据，同时写限制。

## 不可确认情形

- 版本由变量、范围、快照、profile 或外部管理决定，且无法解析最终版本。
- 归档文件名不含版本，元数据也缺失。
- 组件被重打包或合并，无法确认原始版本。
- 只看到类名或包名，没有版本。
