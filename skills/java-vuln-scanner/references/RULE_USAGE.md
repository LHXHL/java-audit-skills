# 本地组件规则使用说明

只在解释 `references/java-vulnerability.yaml` 命中结果时读取本文件。

## 规则性质

- `java-vulnerability.yaml` 是本仓库内置静态规则库，用于快速发现常见 Java 组件旧版本。
- 规则基于 `artifactId:version` 正则匹配，部分 JAR 文件只能从文件名、`pom.properties` 或 `MANIFEST.MF` 推断组件名和版本。
- 规则命中表示“版本落在本地规则覆盖范围内”，不是“当前项目业务风险真实成立”。
- 规则可能过期、重复、漏报或误报；正式交付要写明需复核官方公告、企业 SCA 或供应商安全基线。
- 扫描脚本会拒绝数字版本 token 的前缀误匹配，例如不能把 `2.5.17` 按只覆盖 `2.5.1` 的规则命中。

## 证据等级

| 来源 | 可信度 | 写法 |
|------|--------|------|
| `WEB-INF/lib/*.jar` 文件名 + `pom.properties` | 较高 | 可作为运行时依赖证据 |
| `pom.xml` / `build.gradle` 直接版本 | 较高 | 需注意 scope 和 profile |
| 父 POM / dependencyManagement | 中 | 需确认子模块是否实际引入 |
| JAR 文件名 | 中低 | 可能改名、shade 或厂商重打包 |
| `MANIFEST.MF` | 中低 | 可能缺失或不规范 |
| 类名/包名 | 低 | 只能说明存在代码片段，不能确定版本 |

## 去重规则

- 同一 `component + version + CVE/规则` 在多个模块出现时，按模块列出来源，但在结论中聚合为一个治理项。
- 同一组件命中多个 CVE 时，列出关键规则命中，但不要把它们写成多个已确认业务漏洞。
- 如果同一 JAR 被源码依赖和部署包同时发现，优先以部署包运行时 JAR 为准，同时保留源码依赖作为补充证据。

## 修复版本写法

- 不要写“安全版本为 X”。
- 如果规则描述中包含建议版本，只写“内置规则建议升级到 X，需复核官方公告或企业 SCA”。
- 如果规则未给建议版本，写“无；需官方公告复核”。
- 不根据记忆补写最新版本。

## 不可确认情形

- 版本是 `${...}`、`latest.release`、`SNAPSHOT`、范围表达式或由 profile 决定，且无法解析最终版本。
- JAR 文件名不含版本，`pom.properties` 和 `MANIFEST.MF` 也缺失版本。
- 组件被 shade 到业务 JAR 中，无法确认原始依赖版本。
- 只看到类名或包名，没有版本。
- 构建文件与部署包版本不一致，无法判断运行时采用哪一份。
