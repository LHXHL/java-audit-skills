# 组件版本命中报告模板

生成 `java-vuln-scanner` 报告前必须读取本文件。模板用于依赖版本证据交付，不用于证明业务风险真实成立，不得诱导生成可复制验证材料、通用评分、具体版本建议或未验证攻击链。

## 硬约束

1. 报告必须且只能包含 6 个编号章节。
2. 编号章节标题必须与模板完全一致：`## 1. 扫描概述`、`## 2. 依赖证据统计`、`## 3. 组件版本命中映射`、`## 4. 触发面待核查`、`## 5. 未命中与限制说明`、`## 6. 审计结论`。
3. 不得新增 `## 0`、`## 7`、额外内部检查、验证材料等额外章节；额外信息必须放入模板已有章节。
4. 状态枚举只允许：`版本命中`、`触发面待核查`、`环境条件待确认`、`不可确认`、`未命中`。
5. 不得输出业务风险成立、已验证或验证成功等业务漏洞结论。
6. 不得输出通用评分、具体版本建议、可复制验证请求、攻击字符串、JNDI 地址、反序列化链对象、恶意 XML、利用脚本或真实攻击链。
7. CVE/规则命中必须能追溯到依赖来源文件、组件名和版本；否则写 `不可确认`。
8. 规则描述中的升级建议不得写入报告；只保留组件、当前版本、CVE/规则命中、来源路径、状态和需复核官方公告或企业 SCA 的限制说明。
9. 版本命中项必须说明需要哪些运行条件或专项 skill 继续确认触发面。
10. 正式报告不得出现测试提示词、模型名称、技能源校验、脚本堆栈、审批/权限信息、命令失败、内部规则编号或额外内部检查。
11. 文件名和生成时间必须使用真实生成时间，不得使用 `000000`、`19700101`、占位时间或固定时间。
12. 写完报告后的对话回复必须只包含两行：第一行 `报告路径：<path>`；第二行只写 `发现组件版本命中，详见报告。`、`未发现组件版本命中，详见报告。` 或 `组件版本命中不可确认，详见报告。` 三选一；不得输出 QA 过程、校验通过、最高等级、命中数量、额外组件清单、漏洞摘要或内部校验信息。
13. 统计数字必须是精确数字；无法精确统计时写 `不可确认` 并说明原因，不得写 `约`、`~`、`大约` 或范围数量。
14. 不得用 `规则 #N`、脚本内部序号或规则库行号解释聚合结果；只描述外部可理解的组件、版本、模块和 CVE/规则名。
15. 表格不得包含占位行、空行、`已合并到上表`、`同上占位` 或“为避免歧义保留”等内容；没有真实条目时写 `无`。
16. 来源文件必须写相对于扫描根目录的完整相对路径或绝对路径，不得用 `...`、`…`、`省略` 等截断写法。
17. 第 2 节统计使用依赖实例口径：`已解析依赖 = 版本命中 + 未命中`；`触发面待核查`、`环境条件待确认`、`不可确认` 是命中项的后续状态统计，不参与已解析依赖加总，且各自不得大于 `版本命中`。
18. 如果第 5 节写各模块未命中数量，合计必须等于第 2 节 `未命中`。
19. 不要估算各模块未命中分布；无法按模块精确计算时，不写模块拆分，只说明“未按模块拆分统计”。
20. 若使用 `scripts/scan_dependencies.py --format json --no-save`，第 2 节统计必须按 JSON 字段填充：`依赖总数/已解析依赖 = total_dependencies`，`版本命中 = matched_dependency_instances`，`未命中 = unmatched_dependency_instances`；`触发面待核查` 等后续状态只能在命中项内部统计，唯一组件版本只能来自 `unique_matched_component_versions` 的说明。
21. 第 3 节 `规则/CVE 命中` 只能逐项引用脚本 JSON 中 `vulnerabilities[].name` 实际返回的规则名或 CVE；不得把同组件的其他公告、记忆中的关联 CVE 或未返回规则合并进报告。
22. 第 4 节 `组件命中` 也只能引用第 3 节已列出的脚本命中，不得扩展新 CVE、别名规则或修补历史。

## 文件命名

```text
{project_name}_vuln_scan_{YYYYMMDD_HHMMSS}.md
```

## 完整模板

````markdown
# 【项目名称】 - 组件版本命中报告

生成时间: 【YYYY-MM-DD HH:MM:SS】
扫描路径: 【项目路径、部署包或依赖文件路径】
使用 skill: java-vuln-scanner

---

## 1. 扫描概述

| 项目 | 信息 |
|------|------|
| 扫描范围 | 【项目路径、模块、WEB-INF/lib、pom/build 文件或 JAR】 |
| 扫描文件类型 | 【pom.xml / build.gradle / JAR / WAR / WEB-INF/lib / MANIFEST / pom.properties】 |
| 依赖来源 | 【源码构建文件 / 部署包 JAR / 二者都有 / 未确认】 |
| 模块数量 | 【数量】 |
| 依赖总数 | 【数量】 |
| 使用规则 | `references/java-vulnerability.yaml` 本地规则库 |
| 规则局限 | 本地静态规则，不代表最新官方公告；需复核官方公告或企业 SCA |

## 2. 依赖证据统计

| 类别 | 数量 | 说明 |
|------|------|------|
| 已解析依赖 | 【数量】 | 成功提取组件名和版本 |
| 版本命中 | 【数量】 | 命中本地 CVE/组件规则的依赖实例数；唯一组件版本数可在本说明中补充 |
| 触发面待核查 | 【数量】 | 需要路由、配置、sink 或运行条件确认 |
| 环境条件待确认 | 【数量】 | 依赖 JDK、容器、配置、scope 或部署方式 |
| 不可确认 | 【数量】 | 版本、来源、groupId 或运行时范围不足 |
| 未命中 | 【数量】 | 已解析但未命中本地规则的依赖实例数 |

## 3. 组件版本命中映射

| 序号 | 组件 | 当前版本 | 来源文件 | 所属模块 | 规则/CVE 命中 | 后续确认 | 状态 |
|------|------|----------|----------|----------|---------------|--------------|------|
| 【序号】 | 【artifactId 或 groupId:artifactId】 | 【版本】 | 【pom/build/JAR/MANIFEST/pom.properties 路径】 | 【模块】 | 【仅填写脚本 JSON `vulnerabilities[].name` 返回的规则/CVE；同组件其他 CVE 不得补写】 | 【需复核官方公告或企业 SCA；需补充触发面/环境条件】 | 【版本命中/触发面待核查/环境条件待确认/不可确认】 |

## 4. 触发面待核查

| 序号 | 组件命中 | 可能相关触发面 | 当前项目证据 | 缺失证据 | 建议交接 |
|------|----------|----------------|--------------|----------|----------|
| 【序号】 | 【组件 + CVE/规则】 | 【日志输入/JSON 解析/XML 解析/上传/Struts Action/Shiro RememberMe/Spring MVC 等】 | 【已观察到的配置、依赖、路由、类名；无则写“未观察到”】 | 【需要补充的源码、配置、调用链、JDK/容器、鉴权】 | 【java-route-tracer/java-auth-audit/java-deserialization-audit/java-xxe-audit/java-file-upload-audit/其他/无】 |

## 5. 未命中与限制说明

| 项目 | 说明 |
|------|------|
| 未命中范围 | 【哪些依赖已解析但未命中；数量或模块说明】 |
| 未覆盖范围 | 【未解析到版本的 JAR、shade 包、厂商改名包、缺父 POM、缺 lockfile 等】 |
| 运行时限制 | 【scope 未确认、部署包与源码版本不一致、容器/JDK/配置未知等】 |
| 规则局限 | 本报告仅基于本地规则库，不保证覆盖最新 CVE，不替代官方公告或企业 SCA |
| 非漏洞确认 | 版本命中不等于业务风险真实成立，不构成漏洞确认 |

## 6. 审计结论

| 项目 | 结论 |
|------|------|
| 是否发现组件版本命中 | 【是/否/不可确认】 |
| 最高处理优先级 | 【Critical/High/Medium/Low/无；来自本地规则等级，不是通用漏洞评分】 |
| 主要治理事项 | 【配置核查、触发面专项确认、依赖来源补证、官方公告或企业 SCA 复核；不得写具体版本建议】 |
| 需要补充的证据 | 【官方公告、SCA 结果、JDK/容器、路由、调用链、鉴权、源码/部署包版本】 |
| 建议交接 | 【交给哪些专项 skill；无则写“无”】 |
````
