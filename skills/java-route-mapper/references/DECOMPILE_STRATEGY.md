# 反编译策略参考

只在源码不完整、入口实现或参数对象只有编译产物、依赖包中包含项目入口、或动态分发需要枚举目标时读取本文件。

## 何时反编译

必须反编译：

- `ENTRY_ROOT` 或 `HANDLER_CLASS` 只有编译产物。
- 动态 `DISPATCH_RULE` 需要枚举 `ENTRY_OPERATION`。
- 服务契约需要方法签名和参数结构。
- 参数对象源码缺失，字段无法确定。
- dispatcher 通过分支、映射表、注册表或动态调用分发，源码不可读。

不需要反编译：

- 源码已完整且可读。
- 仅为了美化输出或确认普通语法。
- 非项目入口、非项目参数类型、非项目分发代码。

## 输出目录规则

| 模式 | 反编译目录 |
|------|------------|
| Standalone | `{output_path}/decompiled/` 或用户指定目录 |
| Pipeline worker | `{output_path}/decompiled/agent-1-{N}/`，禁止写其他 worker 目录 |
| Pipeline later stages | 可读写 `{output_path}/decompiled/cache/`，按流水线约束执行 |

## 最小化原则

1. 先反编译入口实现。
2. 从入口实现发现参数对象、父类、接口、分发目标。
3. 只继续反编译必要类。
4. 记录反编译来源路径，方便下游复核。

不要一上来反编译整个项目，除非输入完全是编译产物且无法定位入口。

## 需要提取的信息

| 类型 | 提取内容 |
|------|----------|
| 入口实现 | 元数据、方法签名、协议方法、路径片段、参数 |
| 动作实现 | 业务方法、字段、绑定方法、父类字段 |
| 参数对象 | 字段名、类型、嵌套对象、集合泛型、枚举 |
| 服务契约 | operation、参数名、返回类型 |
| dispatcher | 分支键、字符串常量、映射键、动态目标 |

## 反编译失败时

1. 尝试从接口、父类、配置、契约、调用处补充。
2. 记录失败的编译产物路径和错误原因。
3. 对无法确认的类型标注 `unknown`，不要编造。
4. 如果失败导致无法枚举入口，写 blocked，不输出不完整清单冒充完整。

## Gotchas

- 反编译可能丢失参数名；优先用调试信息、接口、元数据、契约或调用构造字段恢复。
- 生成方法不等于请求参数；字段和绑定逻辑才是参数来源。
- 继承生命周期方法要排除，不能当作入口。
- 编译产物项目仍要遵守精确计数，不允许估算或尾随加号。
